## Immediate — Security + Broken Features

### Task #1 · [S1] Remove JWT secret hardcoded fallback

**Problem**
`backend/src/middleware/authenticate.ts:31` uses `process.env.JWT_SECRET || 'fallback-secret'`. If the `JWT_SECRET` environment variable is ever missing (misconfigured deployment, missing `.env` file, env injection failure), every token is signed and verified against the public string `'fallback-secret'`. An attacker who knows this string can forge valid JWT tokens for any user, including admins, with no credentials at all.

**Methodology**
1. Remove the `|| 'fallback-secret'` fallback so `jwt.verify` receives `undefined` and throws on a missing secret.
2. Add a startup guard in `backend/src/index.ts` that calls `process.exit(1)` with a fatal log if `JWT_SECRET` is not set — fail fast at boot rather than silently accepting forged tokens.
3. Do the same check for `JWT_REFRESH_SECRET`.

**UX impact**
None for end users. Ops impact: the server will refuse to start without the env var, which is the intended behavior.

### Task #2 · [S2/S3/S4/S5] Overhaul RBAC — new role structure, feature permission system, and user creation UI

> **Scope expanded** from the original audit based on product requirements: the platform needs 4 named accounts across 2 distinct privilege tiers, and admins must be able to assign granular per-feature access at invite time.

#### 2a · Account Structure & Role Redesign

**Problem (original)**
The original `requireRole` gaps:
- **S2** — `users.routes.ts:18–114`: All user management mutations are only behind `authenticate`. A VIEWER can call `POST /api/users` and create or delete other users.
- **S3** — `users.routes.ts:126`: Invite endpoint hardcodes `role: 'ADMIN'` — any authenticated user can grant admin access to anyone.
- **S4** — `notifications.routes.ts:18`: `POST /trigger-scan` unguarded — any user can spam the Anthropic API.
- **S5** — `settings.routes.ts:29–76`: `PATCH /settings/org` and `POST /settings/cache/clear` unguarded.

**Product requirement**
The platform has exactly 4 platform user accounts organized in 2 tiers:

| # | Who | Role | Rationale |
|---|-----|------|-----------|
| 1 | Developer | `ADMIN` | Full technical + business access; can modify platform settings, manage all users |
| 2 | Senior HR Manager | `ADMIN` | Full business access; same top-level privilege as developer account |
| 3 | HR Staff Employee A | `HR_STAFF` | Does most day-to-day work in the app; lenient default access |
| 4 | HR Staff Employee B | `HR_STAFF` | Same as above |

The current `VIEWER` role is too restrictive for HR staff who do operational work. The current `HR_MANAGER` role name is ambiguous now that the Senior HR Manager holds ADMIN. Rename/repurpose:
- `VIEWER` → retired or demoted to true read-only external access
- `HR_MANAGER` → retained as an intermediate role for external HR Managers if added later
- `HR_STAFF` → new role, added to the Prisma schema and shared enum

**Role capability matrix**

| Capability | ADMIN | HR_STAFF |
|------------|-------|----------|
| View all data modules | ✅ | ✅ (default) |
| Create / edit employees | ✅ | ✅ (default) |
| Delete employees | ✅ | configurable |
| Run scenario previews | ✅ | ✅ (default) |
| Apply scenarios (org-wide salary write) | ✅ | ❌ (ADMIN override only) |
| Manage benefits enrollments | ✅ | ✅ (default) |
| Import data (Data Center) | ✅ | ✅ (default) |
| Trigger AI scan | ✅ | ❌ (configurable) |
| Send platform emails | ✅ | ✅ (default) |
| Invite / manage users | ✅ | ❌ |
| Edit org settings / clear cache | ✅ | ❌ |
| View audit log | ✅ | ❌ (configurable) |

**Methodology — backend**
1. Add `HR_STAFF = 'HR_STAFF'` to Prisma `UserRole` enum → run migration.
2. Add `requireRole('ADMIN')` to all mutating routes in `users.routes.ts` (S2 fix).
3. Accept `role` from request body in invite endpoint; validate against `UserRole`; never hardcode (S3 fix).
4. Add `requireRole('ADMIN')` to `POST /trigger-scan` (S4 fix).
5. Add `requireRole('ADMIN')` to `PATCH /settings/org` and `POST /settings/cache/clear` (S5 fix).
6. Update all `requireRole` calls that previously listed `'HR_MANAGER'` to also accept `'HR_STAFF'` where appropriate (e.g. import, scenario run).

**UX impact**
HR Staff users can access the full operational surface of the app by default. ADMIN-only destructive actions (scenario apply, settings, user management) are hidden from HR_STAFF in both the UI and enforced at the API layer.

#### 2b · Feature Permission System (per-user granular access)

**Problem**
Even with roles, the HR Manager needs to be able to customize exactly which features each HR Staff employee can access when creating their account — without giving them a different role. E.g. one HR Staff member handles imports and benefits; another handles performance and AI insights only. A binary role system cannot express this.

**Design**
Add a `permissions` JSON field (string array) to the `User` model. Each entry is a feature key. When `permissions` is `null`, the role's default access applies. When it contains entries, they take precedence for that user.

**Feature keys (complete list)**

| Key | Default for HR_STAFF | Description |
|-----|---------------------|-------------|
| `dashboard` | ✅ | Dashboard view |
| `employee.view` | ✅ | Employee directory + profiles |
| `employee.manage` | ✅ | Create, edit employees |
| `employee.delete` | ✅ | Delete employees |
| `pay_equity` | ✅ | Pay equity analysis |
| `salary_bands` | ✅ | Salary band designer (view + edit) |
| `scenario.view` | ✅ | View scenario list |
| `scenario.run` | ✅ | Run / preview scenarios |
| `scenario.apply` | ❌ | Apply scenarios — org-wide salary write |
| `benefits.view` | ✅ | Benefits catalog + enrollments |
| `benefits.manage` | ✅ | Enroll employees, edit benefits |
| `variable_pay` | ✅ | Variable pay analytics |
| `performance.view` | ✅ | View performance ratings |
| `performance.manage` | ✅ | Create / edit ratings |
| `ai_insights` | ✅ | AI insights module + chat |
| `ai_scan` | ❌ | Trigger proactive AI anomaly scan |
| `data_center` | ✅ | Data Center imports / exports |
| `notifications` | ✅ | Notifications center |
| `email` | ✅ | Send platform emails |
| `audit_log` | ❌ | View audit log |
| `user.manage` | ❌ | Invite and manage platform users |
| `settings.platform` | ❌ | Platform settings + cache management |

ADMIN accounts bypass the `permissions` check entirely — their access is unconditional.

**Methodology — backend**
1. Add `permissions String[] @default([])` to `User` model in Prisma schema → run migration.
2. Include `permissions` in the JWT payload and in the `/api/auth/me` response.
3. Create a `requireAccess(feature: string)` middleware:
```typescript
// backend/src/middleware/requireAccess.ts
export const requireAccess = (feature: string) =>
  (req: Request, res: Response, next: NextFunction) => {
    const user = req.user as JwtPayload;
    if (user.role === 'ADMIN') return next(); // ADMIN bypasses all
    const defaults = HR_STAFF_DEFAULT_PERMISSIONS; // from shared/constants
    const allowed = user.permissions?.length
      ? user.permissions          // explicit override set at invite time
      : defaults;                 // fall back to role defaults
    if (allowed.includes(feature)) return next();
    res.status(403).json({ error: { code: 'FORBIDDEN', message: 'Access denied. Insufficient permissions.' } });
  };
```
4. Apply `requireAccess('scenario.apply')` to the scenario apply route, replacing the bare `requireRole('ADMIN')` — now HR_STAFF with explicit permission can apply too if the admin grants it.
5. Add `permissions` field to `UserInvite` model so it travels through the invite flow and is written to `User` on accept.

**Methodology — frontend: User creation / invite modal**

The existing invite modal needs a complete rebuild into a two-step form:

*Step 1 — Identity & Role*
- Email address input
- Role selector: `ADMIN` | `HR_STAFF` (dropdown, read-only labels explain each)
- If ADMIN is selected, skip to confirmation (no feature toggles needed)

*Step 2 — Feature Access (HR_STAFF only)*
- Section header: "Feature Access — defaults are pre-selected based on HR Staff role. The HR Manager can adjust before sending the invite."
- Feature toggles grouped by module:
  - **Employee Management**: View, Manage, Delete
  - **Compensation**: Pay Equity, Salary Bands
  - **Scenarios**: View, Run, Apply *(Apply has a warning badge: "High impact — writes salaries org-wide")*
  - **Benefits & Equity**: View, Manage
  - **Analytics**: Variable Pay, Performance View, Performance Manage
  - **AI**: AI Insights & Chat, Trigger Anomaly Scan *(Scan badge: "Costs Anthropic API calls")*
  - **Operations**: Data Center, Email, Notifications
  - **Admin**: Audit Log *(disabled by default, Admin can enable)*
- "Reset to defaults" button pre-populates HR_STAFF defaults
- "Grant all" / "Revoke all" shortcuts

*Confirmation step*
- Shows a summary card: name, role, and an abbreviated permission list ("Can do X, Y, Z · Cannot do A, B")
- "Send Invite" button

**Methodology — frontend: runtime access guards**

Add a `useAccess(feature: string): boolean` hook in `frontend/src/hooks/useAccess.ts`:
```typescript
export const useAccess = (feature: string): boolean => {
  const { user } = useAuthStore();
  if (user?.role === 'ADMIN') return true;
  const perms = user?.permissions ?? HR_STAFF_DEFAULT_PERMISSIONS;
  return perms.includes(feature);
};
```

Use this hook to conditionally render sensitive controls in every page:
- "Apply Scenario" button in `ScenarioModelerPage.tsx` → `useAccess('scenario.apply')`
- "Trigger AI Scan" in `NotificationsCenterPage.tsx` → `useAccess('ai_scan')`
- "Delete Employee" in `EmployeeDirectoryPage.tsx` → `useAccess('employee.delete')`
- "Invite User" in user management → `useAccess('user.manage')` (ADMIN only in practice)
- Platform Settings tab in sidebar → `useAccess('settings.platform')`

**UX impact**
- Admins creating a new HR Staff invite see a feature toggle panel pre-populated with sensible defaults. They can customize before sending — no need to edit the account after the fact.
- HR Staff users only see sidebar items and action buttons they have access to. Unauthorized feature links are hidden (not just disabled), reducing cognitive noise.
- If the HR Manager later needs to change an HR Staff member's access, a new "Edit Permissions" panel on the Users page lets them update the `permissions` array and re-send an updated token or apply immediately.

### Task #3 · [L3/T3] Fix RSU P1 eligibility threshold

**Problem**
`rsu.service.ts:45–47` rejects employees at band P1 from RSU grants. The eligibility check uses `bandIdx < BAND_ORDER.indexOf('P2')` (index 3), which excludes P1 (index 2). CLAUDE.md and business rules say **P1 and above are eligible** — A1/A2 are the only ineligible bands. Simultaneously, `shared/constants/index.ts:70–74` sets `RSU_ELIGIBILITY.MIN_BAND_LEVEL: 4`, which corresponds to P2 rather than P1.

These two mismatched values mean every P1 employee is silently denied RSU grants they are entitled to.

**Methodology**
1. In `rsu.service.ts`: change the threshold comparison to `bandIdx < BAND_ORDER.indexOf('P1')` (index 2).
2. In `shared/constants/index.ts`: change `MIN_BAND_LEVEL` from `4` to `2`.
Both must change together — they represent the same business rule in two places.

**UX impact**
P1 employees will start appearing in the RSU Grants tab under Benefits Management. Previously their grants would not be generated or shown.

### Task #5 · [F4] Fix employee delete query invalidation key shape

**Problem**
`EmployeeDirectoryPage.tsx:46` — after a successful employee delete, the `onSuccess` callback calls:
```typescript
queryClient.invalidateQueries({ queryKey: ['employees', {}] })
```
React Query matches invalidations by prefix. The key `['employees', {}]` does not match `['employees', filters]` (where `filters` is an actual filter object from state). The result is the employee list does not refetch after deletion — the deleted employee remains visible until a manual page refresh.

**Methodology**
Change the invalidation to:
```typescript
queryClient.invalidateQueries({ queryKey: ['employees'] })
```
This uses the prefix form and matches all employee queries regardless of filter state.

**UX impact**
After deleting an employee, the directory list now immediately removes the deleted row without requiring a page refresh.

### Task #6 · [L1] Fix scenario apply: cache bust, socket emit, annualCtc, derived fields

**Problem**
`scenarios.service.ts:168–196` — `apply()` bulk-writes new salaries to employee records but:
1. Does not bust any Redis cache — stale salary data is served from cache for up to the TTL duration after apply.
2. Does not emit any socket event — all connected clients stay on the old data.
3. Does not update `annualCtc` — the total cost field stays at the pre-scenario value.
4. Does not recalculate `compaRatio` — pay equity ratios are wrong until a separate trigger.

A scenario "applied" through the UI looks like it worked (the mutation succeeds) but the rest of the application continues showing pre-scenario data until a cold restart or cache expiry.

**Methodology**
After the `apply()` loop completes:
1. Call `cacheDelPattern('dashboard:*')` and `cacheDelPattern('employees:*')` to bust derived caches.
2. Call `emitEmployeeDataChanged()` and `emitDashboardRefresh()` from `lib/socket.ts`.
3. For each affected employee, recalculate `annualCtc` (base + variable + bonus) and `compaRatio` (salary ÷ band midpoint) and write back to the DB in the same transaction or a follow-up batch.

**UX impact**
After clicking "Apply Scenario", the dashboard KPIs, employee salary figures, pay equity scores, and compa-ratio heatmaps all update in real time across all connected browser sessions without requiring a refresh.

### Task #7 · [S6] Add socket auth token validation on handshake

**Problem**
`backend/src/lib/socket.ts:42–48` — Socket.io accepts connections with no token check. Any unauthenticated client (or a logged-out browser tab that kept a stale connection) receives all broadcast events: `NOTIFICATION_NEW`, `PAY_ANOMALY`, `EMPLOYEE_DATA_CHANGED`, etc. This leaks compensation data and notification content to anyone who can reach the socket endpoint.

**Methodology**
Add `io.use(...)` middleware that runs before the connection is accepted:
```typescript
io.use((socket, next) => {
  const token = socket.handshake.auth?.token as string | undefined;
  if (!token) return next(new Error('Authentication required'));
  try {
    const payload = jwt.verify(token, process.env.JWT_SECRET!) as JwtPayload;
    (socket as any).user = payload;
    next();
  } catch { next(new Error('Invalid token')); }
});
```
On the frontend (`frontend/src/lib/socket.ts`), pass the access token in the connection options:
```typescript
{ auth: { token: sessionStorage.getItem('accessToken') } }
```

**UX impact**
No visible change. Unauthenticated connections are silently rejected at the handshake layer.

### Task #8 · [S7] Sanitize email HTML in SentMailsPage to fix stored XSS

**Problem**
`SentMailsPage.tsx:86` renders email body HTML via `dangerouslySetInnerHTML` with no sanitization. If any sent email contains a `<script>` tag or an event handler attribute (e.g. from a compromised email template or an injection in a `Subject` field), it executes in the context of the logged-in user's session — a classic stored XSS attack.

**Methodology**
Option A (preferred): Install `dompurify` + `@types/dompurify` and sanitize before render:
```typescript
import DOMPurify from 'dompurify';
{ __html: DOMPurify.sanitize(emailHtml) }
```
Option B: Render the HTML inside a sandboxed `<iframe srcdoc={emailHtml} sandbox="allow-same-origin" />` to prevent script execution.

**UX impact**
Emails render identically for well-formed content. Malicious script tags are stripped silently — no user-visible change in normal operation.

### Task #9 · [L4] Fix aiInsights filtersHash not used in DB lookup

### Task #16 · [L2] Fix scenario run overwriting original snapshot data

**UX impact**

### Task #18 · [L6] Fix non-unique employeeId generation in AddEmployeeModal

**Problem**
`AddEmployeeModal.tsx:127` generates a client-side `employeeId` using `Date.now().toString().slice(-6)`. This produces a 6-digit numeric string that:
- Collides with the CSV-seeded `EMP001–EMP200` range if the number happens to match.
- Collides between two browser sessions creating employees at the same millisecond (concurrent use).
The backend's unique constraint on `employeeId` will throw a 500 for the second insert, with no helpful user-facing error.

**Methodology**
Remove client-side `employeeId` generation entirely. Let the backend assign a UUID or sequential ID in the service layer. If the frontend form requires a display ID before server round-trip, use `crypto.randomUUID()` as a temporary placeholder that gets replaced on the API response.

**UX impact**
No visible change in normal use. Eliminates silent 500 errors in concurrent scenarios.

### Task #22 · [N1/N2/N3] Fix N+1 query problems in import, benefits, and scenario services

**Problem**
Three services run sequential per-row DB queries inside loops:

- **N1** — `import.service.ts:525–530`: `buildEmployeeData` calls `prisma.employee.findFirst` for each row in an import file — up to 1000 sequential DB round-trips for a full import.
- **N2** — `benefits.service.ts:210`: `importUtilizationData` similarly calls `findFirst` per row during benefits utilization import.
- **N3** — `scenarios.service.ts:181–193`: `apply()` calls `prisma.employee.update` per employee in a loop — 200 sequential writes for a full-org scenario apply.

These cause imports and scenario applies to be disproportionately slow (seconds to minutes for larger files) and put excessive load on the Neon DB.

**Methodology**
N1/N2 fix — pre-load a lookup Map before the loop:
```typescript
const employees = await prisma.employee.findMany({
  select: { id: true, employeeId: true, email: true }
});
const employeeMap = new Map([
  ...employees.map(e => [e.employeeId.toLowerCase(), e.id] as const),
  ...employees.map(e => [e.email.toLowerCase(), e.id] as const),
]);
// Replace findFirst(where: employeeId) with employeeMap.get(rawId)
```

N3 fix — batch updates using `prisma.$transaction([...updates])` instead of sequential awaits.

**UX impact**
Employee CSV imports and benefits utilization imports complete significantly faster. Scenario apply responds faster, especially for large orgs. Users see less "pending" spinner time on the Data Center and Scenario Modeler pages.

### Task #23 · [S8/S9/S10] Fix email plaintext password, role leak in 403, Redis error handling

**Problem**
Three separate security/reliability issues:

- **S8** — `email.service.ts:75`: Plaintext passwords are embedded in email bodies and transmitted in HTTP request bodies. If email is intercepted or the email provider is compromised, credentials are exposed in cleartext.
- **S9** — `requireRole.ts:19`: The 403 error message reads something like `"Required role: ADMIN, your role: VIEWER"`. This leaks the authenticated user's role to potential attackers probing the API.
- **S10** — `redis.ts:24–47`: Cache helper functions (`cacheGet`, `cacheSet`, `cacheDel`) have no try/catch. A Redis connection failure (Upstash outage, TLS error) causes unhandled promise rejections that bubble up as HTTP 500s across the entire application.

**Methodology**
- **S8**: Remove the password-sending code path. Use the existing invite/password-reset token flow — send a reset link, never the password itself.
- **S9**: Change the 403 message to a generic `'Access denied. Insufficient permissions.'`
- **S10**: Wrap each cache helper in try/catch: `cacheGet` returns `null` on error (triggers a cache miss path), `cacheSet`/`cacheDel` log the error and continue silently.

**UX impact**
- S8: Users receive a reset link instead of a password in their email — more secure and already the expected pattern.
- S9: No visible change. Error message is less specific.
- S10: Redis outages no longer take down the entire API — users continue with slower (uncached) responses rather than seeing 500 errors.
